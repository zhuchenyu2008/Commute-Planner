# 历史记录页面规格

## 页面目标

历史记录页用于查看当前、未来、已完成和已取消的通勤规划记录，并支持复用历史目的地快速再次规划。它帮助用户回看 Agent 做过的决策，同时避免首页承载过多记录管理功能。

## 用户任务

- 查看当前和未来的行程。
- 回看已完成或已取消的历史规划。
- 按目的地、日期、路线类型和状态筛选记录。
- 点击记录进入详情页。
- 复用历史目的地再次规划。
- 清理不再需要的历史记录。

## 全局约定

- 未登录访问历史记录时跳转登录门禁，登录成功后返回历史页。
- 底部导航展示：首页、历史、设置；历史 tab 高亮。
- 详情页从历史记录进入时，返回按钮回到历史页并保留筛选条件。
- 清理历史默认软删除，不直接物理删除数据。
- 历史页使用列表和分组，不使用大面积统计卡片。

## 布局结构

1. 顶部标题区
   - 标题：“历史记录”。
   - 次要文案：“查看过去的路线和提醒”。
   - 可选右侧操作：清理。

2. 搜索与筛选区
   - 搜索框：目的地、地点别名、路线名。
   - 日期筛选：全部、今天、未来、本周、自定义。
   - 状态筛选：全部、进行中、未来、已完成、已取消。
   - 路线筛选：公交/地铁、骑行、步行、组合路线。

3. 当前/未来行程分组
   - 展示活跃行程和计划中的行程。
   - 行程状态醒目，便于快速进入详情。

4. 历史行程分组
   - 按日期倒序展示。
   - 同一天多条记录按创建时间倒序。

5. 记录项
   - 目的地名称。
   - 到达日期和时间。
   - 路线摘要。
   - 最晚出发时间。
   - 状态和提醒通道。
   - 操作：查看、再次规划、删除。

6. 空状态
   - 无记录时显示轻量说明和“去规划”按钮。
   - 搜索无结果时显示“清除筛选”按钮。

## 页面状态

- `unauthenticated`：跳转登录页。
- `loading`：加载历史记录。
- `readyWithTrips`：存在记录。
- `empty`：没有任何记录。
- `emptyFiltered`：筛选后没有结果。
- `deleting`：正在软删除记录。
- `duplicatePlanning`：正在基于历史记录创建新规划草稿。
- `offline`：后端不可达。
- `authExpired`：认证过期，跳转登录页。

## 核心交互

- 页面加载时调用 `GET /api/trips`。
- 搜索和筛选优先在服务端执行；输入搜索词时做短延迟请求，避免频繁调用。
- 点击记录主体进入 `/trips/{id}`。
- 点击“再次规划”调用 `POST /api/trips/{id}/duplicate`，后端返回规划草稿或首页输入参数。
- 再次规划成功后跳转首页，并把目的地和偏好带入输入框等待用户确认。
- 点击删除时显示确认弹窗；确认后调用 `DELETE /api/trips/{id}`，列表中移除该项。
- 删除为软删除，后端可保留审计数据和 Agent 上下文。

## 接口

### `GET /api/trips`

查询参数：

```text
q=龙湖天街
status=active,scheduled,completed,cancelled
routeType=transit,bike,walk,mixed
dateFrom=2026-06-26
dateTo=2026-07-03
includeDeleted=false
```

### `GET /api/trips/{id}`

用于进入详情页时加载单条行程。

### `POST /api/trips/{id}/duplicate`

基于历史目的地、到达时间模式和路线偏好创建再次规划草稿。

响应示例：

```json
{
  "draftText": "明天 9:15 到龙湖天街",
  "destinationText": "龙湖天街",
  "routePreference": "mixed"
}
```

### `DELETE /api/trips/{id}`

软删除历史记录。活跃行程删除前应要求先停止监控。

## 验收点

- 历史页能区分当前/未来行程与过去行程。
- 点击任何记录都能进入对应详情页。
- 再次规划不会直接创建提醒，必须回到首页或 Agent 流程让用户确认。
- 删除历史记录默认软删除，活跃行程需要先停止监控。
- 搜索和筛选状态在进入详情再返回后保持不变。
